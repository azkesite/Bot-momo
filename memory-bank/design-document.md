# 聊天机器人产品设计文档

## 1. 产品概述

### 1.1 产品名称
群聊拟人聊天机器人

### 1.2 产品目标
使用 TypeScript 构建一个运行在群聊场景中的聊天机器人，让它在发言节奏、上下文理解、人物记忆和表达方式上尽可能接近真实群友，而不是一个“每条都回、每次都回很长”的工具型机器人。

### 1.3 核心体验目标
- 像群成员，而不是客服
- 能理解“是不是在跟它说话”
- 有记忆，能记住不同群友的特点和历史话题
- 发言有分寸，不刷屏，不抢戏
- 被点名时必须有回应

## 2. 需求范围

### 2.1 必须满足的需求
1. 被 `@` 时必须回复
2. 不需要每句话都回复，只有在与机器人相关、群友明显在和机器人对话、或命中触发规则时才回复
3. 回复时可以拆成多句发送，而不是一次发送整段大文本
4. 为每个群友维护独立记忆点，记住其说过的话、偏好、称呼、常聊主题或关系上下文
5. 支持配置关键词，命中关键词时触发回复

### 2.2 非目标
- 不追求全知全能问答
- 不追求每次都长篇高质量输出
- 不强行参与所有群聊话题
- 不在第一阶段做复杂社交关系推理，如派系分析、情绪操控等

## 3. 使用场景

### 3.1 典型场景
- 群友 `@机器人` 提问、调侃、点名
- 群友直接使用机器人的名字或常用昵称和它说话
- 群内讨论到机器人熟悉的话题，且与其历史发言内容相关
- 群友说到预设关键词，例如游戏、项目名、约定暗号、固定梗
- 群友延续上一轮和机器人的上下文继续聊天

### 3.2 不应频繁插话的场景
- 群友之间明显在互相聊天，与机器人无关
- 高频闲聊刷屏场景
- 敏感争论中无明确召唤
- 机器人刚刚已经回复过，短时间内不应再次抢话

## 4. 产品原则

### 4.1 拟人优先
机器人应尽量表现出“观察、理解、选择性发言”的特征，而不是“监听到消息就立刻响应”。

### 4.2 低打扰
发言频率要受控。宁可少说，也不要让群友感觉像自动回复脚本。

### 4.3 连续人格
机器人要对同一个群友保持相对稳定的称呼、语气和记忆连续性。

### 4.4 可配置
触发阈值、冷却时间、关键词、人格风格、记忆保留规则都应支持配置。

## 5. 核心功能设计

## 5.1 消息响应决策引擎

### 目标
决定“这条消息该不该回”。

### 输入
- 当前消息内容
- 发送者 ID
- 群 ID
- 是否 `@机器人`
- 是否引用/回复机器人消息
- 是否包含机器人名字、昵称、别名
- 是否命中关键词
- 最近 N 条群聊上下文
- 机器人最近发言时间
- 当前群聊活跃度

### 输出
- `must_reply`：必须回复
- `should_reply`：建议回复
- `skip`：跳过
- `delay_reply`：延迟回复

### 规则优先级
1. `@机器人`：必须回复
2. 明确对话：如“你怎么看”“bot你人呢”“刚才问你那个”，高优先级回复
3. 引用机器人上一条消息继续追问：高优先级回复
4. 命中关键词：中优先级回复
5. 与机器人强相关的话题：中优先级回复
6. 普通群友互聊且与机器人无关：默认不回复

### 建议增加的决策因子
- 冷却时间：例如 20 秒内已回复过，则提高跳过概率
- 连续发言限制：连续 2 次发言后进入沉默窗口
- 群聊热度抑制：消息密度过高时降低参与概率
- 用户召回权重：高频和机器人互动的群友，可略微提高回复概率

## 5.2 `@` 触发机制

### 规则
- 只要被 `@`，必须有响应
- 即便无法完整回答，也应先做兜底回应，例如：
  - “我在”
  - “看到了，等我一下”
  - “这个我想想”

### 异常策略
- 模型超时：先发短句占位，再补答
- 上下文不足：先追问澄清，而不是胡乱回答
- 多人同时 `@`：按时间顺序排队处理，可合并相近问题

## 5.3 选择性回复机制

### 目标
避免“每句话都接”，保持像真人一样的参与感。

### 识别策略
- 明确称呼机器人
- 语义上是在询问机器人观点
- 话题和机器人刚才的发言强相关
- 群友在延续与机器人的私有上下文

### 频率控制策略
- 设置基础回复概率
- 设置群级冷却时间
- 设置用户级冷却时间
- 设置话题级冷却时间
- 高活跃群中降低主动插话频率

### 示例
- 群友 A：`@bot 晚上打不打游戏`
  - 必回
- 群友 B：`昨天你不是说会写完吗`
  - 若结合上下文判断是在对机器人说，可回
- 群友 C：`我去吃饭了`
  - 默认不回

## 5.4 分句发送机制

### 目标
降低机器人“一次性输出完整大段文字”的机械感。

### 行为设计
- 一次生成完整回复后，根据语义拆成 2 到 4 句
- 按短时间间隔逐条发送
- 间隔可以带少量随机抖动，例如 600ms 到 2500ms
- 短句优先，口语优先，避免每句都太完整

### 示例
原始回复：
“这个方案可以做，不过我建议先把记忆模块和回复策略拆开，不然后面很容易耦合。”

拆分后发送：
1. “这个方案可以做”
2. “不过我建议先把记忆模块和回复策略拆开”
3. “不然后面会很容易耦合”

### 约束
- 不要拆得过碎，避免像网络延迟
- 不要每次都拆句，保留自然变化
- 严肃问题或指令型回复可整句发送

## 5.5 群友记忆系统

### 目标
让机器人对每个群友形成持续印象，表现得像“真的在群里待过”。

### 记忆对象
每个群友单独维护一份记忆档案。

### 记忆内容建议
- 用户基础信息：`userId`、昵称、常用称呼
- 显性偏好：喜欢的话题、游戏、口头禅
- 历史事实：说过的关键事件、计划、状态变化
- 互动关系：和机器人聊过哪些内容、熟悉程度
- 风格标签：爱开玩笑、爱吐槽、喜欢技术讨论等
- 最近上下文摘要：最近几次对话主题和结论

### 记忆分层
- 短期记忆：最近 20 到 50 条相关对话
- 中期记忆：近期稳定话题与事实摘要
- 长期记忆：高价值人设信息与关系信息

### 更新策略
- 新消息进入后，先判断是否值得写入记忆
- 高价值信息写入结构化记忆
- 低价值闲聊只保留在短期上下文
- 定期压缩摘要，避免上下文无限膨胀

### 示例
- “我下周要考研” -> 长期记忆候选
- “我今天在加班” -> 中期记忆候选
- “哈哈哈哈” -> 不写长期记忆

## 5.6 上下文管理

### 目标
让机器人知道“当前这轮聊天在说什么”，同时不丢掉对不同群友的记忆。

### 建议结构
- 群公共上下文
  - 最近消息窗口
  - 当前热议主题
  - 最近一次机器人发言
- 用户私有上下文
  - 与该用户最近的对话摘要
  - 用户长期记忆
- 触发上下文
  - 此次为什么回复
  - 回复目标是谁
  - 当前回复应保持什么语气

### 技术建议
- 原始消息和摘要并存
- 优先检索“与当前发言者相关”的历史
- 再补充群级公共上下文
- 对模型输入做裁剪，保证 prompt 可控

## 5.7 关键词触发系统

### 目标
在没有直接 `@` 的情况下，通过关键词让机器人自然出现。

### 功能要求
- 支持配置多个关键词
- 支持同义词和别名
- 支持精确匹配与模糊匹配
- 支持不同关键词对应不同回复策略

### 关键词分类建议
- 唤醒类：机器人名字、昵称、别称
- 话题类：项目名、游戏名、角色名、固定梗
- 事件类：开会、上线、抽卡、开黑
- 情绪类：无聊、崩溃、睡不着

### 命中后的处理
- 命中不代表一定回复
- 还需结合冷却时间、最近是否已发言、当前语境是否适合介入
- 对高优先级关键词可直接触发

## 6. 拟人化设计

## 6.1 语言风格
- 用词口语化，不要全是标准书面语
- 允许适度语气词，如“啊”“吧”“我感觉”
- 避免每次都用完整结论式表达
- 可以有犹豫感，如“我想想”“感觉像是”

## 6.2 节奏模拟
- 不秒回所有消息
- 回复延迟可根据消息长度、复杂度动态变化
- 有时先发短句，再补一句
- 有时不接话，模拟真实群友潜水

## 6.3 记忆带来的真实感
- 偶尔自然提及群友之前说过的内容
- 使用稳定称呼
- 能承接前几天甚至更久前的话题

## 6.4 避免机械感的负面模式
- 不要每次都“您好，我认为”
- 不要每条都热情回复
- 不要连续输出超长段落
- 不要无差别附和所有人

## 7. 系统模块设计

### 7.1 模块划分
- 消息接入层
- 事件解析器
- 响应决策引擎
- 上下文管理器
- 用户记忆管理器
- LLM 回复生成器
- 分句发送器
- 关键词配置中心
- 风控与频率控制器

### 7.2 推荐处理流程
1. 接收群消息事件
2. 解析基础信息和触发特征
3. 判断是否需要回复
4. 拉取相关上下文和用户记忆
5. 生成候选回复
6. 判断是否需要分句
7. 按节奏发送
8. 回写本轮对话与记忆摘要

## 8. 数据结构建议

### 8.1 消息对象
```ts
type ChatMessage = {
  messageId: string;
  groupId: string;
  userId: string;
  nickname: string;
  content: string;
  timestamp: number;
  mentions?: string[];
  replyToMessageId?: string;
};
```

### 8.2 用户记忆对象
```ts
type UserMemory = {
  userId: string;
  groupId: string;
  nicknameHistory: string[];
  aliases: string[];
  traits: string[];
  preferences: string[];
  keyFacts: string[];
  recentTopics: string[];
  relationshipSummary: string;
  lastInteractionAt?: number;
  updatedAt: number;
};
```

### 8.3 回复决策结果
```ts
type ReplyDecision = {
  action: "must_reply" | "should_reply" | "skip" | "delay_reply";
  reason:
    | "mentioned"
    | "directed_to_bot"
    | "keyword_hit"
    | "topic_related"
    | "cooldown"
    | "not_relevant";
  confidence: number;
  delayMs?: number;
};
```

### 8.4 关键词配置
```ts
type KeywordRule = {
  id: string;
  keyword: string;
  matchType: "exact" | "fuzzy" | "regex";
  priority: number;
  enabled: boolean;
  cooldownMs?: number;
};
```

## 9. 配置项建议

### 基础配置
- 机器人名称
- 机器人昵称/别名
- 默认语气风格
- 单条回复最大长度
- 是否开启分句发送

### 行为配置
- `@` 必回开关
- 主动回复概率
- 群级冷却时间
- 用户级冷却时间
- 关键词触发开关
- 分句条数范围
- 句间发送延迟范围

### 记忆配置
- 每个用户最大记忆条数
- 长期记忆提取阈值
- 摘要压缩周期
- 记忆过期时间

## 10. 风险与约束

### 10.1 主要风险
- 回复过于频繁，像脚本
- 记忆错误，导致“记混人”
- 关键词过多，造成乱入
- 分句发送过度，显得刻意
- 群上下文过长，导致模型成本和延迟上升

### 10.2 控制措施
- 加强回复门控
- 记忆写入前做价值判断
- 对重要事实加入置信度字段
- 设置全局和局部冷却
- 定期总结上下文，避免长 prompt

## 11. 第一阶段 MVP 建议

### MVP 必做
- 接入群消息
- 实现 `@` 必回
- 实现基础“是否回复”判断
- 实现关键词触发
- 实现分句发送
- 实现每个群友独立记忆
- 实现最近上下文摘要

### MVP 可暂缓
- 复杂人格系统
- 情绪状态机
- 多群差异化人格
- 高级关系图谱
- 自动学习新关键词

## 12. 验收标准

### 功能验收
- 被 `@` 后机器人能稳定回复
- 普通群聊中机器人不会句句都回
- 命中关键词时能按规则出现
- 回复能按配置拆成多句发送
- 不同群友拥有独立记忆记录

### 体验验收
- 群友不会明显感觉它在刷存在感
- 群友会感觉它“记得我说过什么”
- 回复节奏比传统机器人更自然
- 长时间运行后上下文不会严重失控

## 13. 后续迭代方向
- 增加人格状态，如困、忙、兴奋、潜水中
- 增加不同群的不同说话风格
- 增加记忆可信度与冲突解决机制
- 增加消息重要度分类器
- 增加更精细的“像人一样的沉默”策略

---

这份文档的核心思想是：机器人不是“看见消息就回答”，而是“像一个长期在群里的成员，先判断要不要开口，再自然地开口”。
