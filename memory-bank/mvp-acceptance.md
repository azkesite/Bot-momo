# MVP 验收报告

## 1. 验收结论

当前版本已经达到本轮初版 MVP 的开发完成标准，可以进入小范围真实群灰度联调阶段。

本轮结论依据：

- `corepack pnpm typecheck` 通过
- `corepack pnpm test` 通过
- `corepack pnpm lint` 通过
- 端到端消息回放测试通过
- 异常与降级测试通过

## 2. 五个核心要求核对

### 2.1 被 `@` 必须回复

结果：通过

证据：

- `must_reply` 规则已接入总决策引擎
- `message-processor.e2e.test.ts` 覆盖了被 `@` 后成功发送回复

### 2.2 不要每句话都回复

结果：通过

证据：

- 相关性判断与总决策引擎已接入
- 普通无关消息在 `message-processor.e2e.test.ts` 中会稳定返回 `skip`

### 2.3 回复可以分几句发送

结果：通过

证据：

- 已实现分句规则、顺序调度和发送防重
- 端到端回放测试验证了长回复会拆句发送

### 2.4 可为每个群友建立记忆

结果：通过

证据：

- 已实现用户记忆、memory facts、关系摘要、最近互动时间回写
- 端到端回放测试验证了偏好信息会进入长期记忆

### 2.5 可通过关键词触发出现

结果：通过

证据：

- 已实现关键词规则存储、缓存、命中判断和强制回复
- 端到端回放测试验证了关键词命中后会发送回复

## 3. 本轮已完成的调试与防回归能力

- 管理接口已可查看健康状态、关键词规则、用户记忆摘要和最近回复日志
- 已建立异常与降级测试：
  - LLM 失败 -> 兜底回复
  - 发送失败 -> 回复日志标记失败并输出错误日志
- 已建立可重复执行的消息回放测试

## 4. 当前已知限制

- 真实 QQ 群联调尚未完成，本轮验收主要基于自动化测试
- NapCat 当前聚焦群消息主链路，未覆盖更多事件类型
- 发送调度仍为进程内顺序执行，未接入正式队列和退避重试
- 默认 LLM transport 仍是启发式最小实现，等待替换真实 provider

## 5. 建议的下一阶段动作

1. 接入真实 OpenAI/Claude/GLM/DeepSeek/Kimi transport。
2. 接入 BullMQ，把发送调度从进程内执行迁到任务队列。
3. 连接真实 NapCat 与 QQ 群，做一轮人工灰度联调。
4. 根据灰度结果收紧主动插话概率、拆句频率和管理接口展示内容。
